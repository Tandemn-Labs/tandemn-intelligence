A machine learning team submits this request: "Run inference on a 2 million row dataset using DeepSeek-V3 (671B parameters, MoE architecture with 256 experts, 8 active per token). Each row has an average input of 800 tokens and we need approximately 200 tokens of output per row. Total: roughly 1.6 billion input tokens and 400 million output tokens. We need this completed within 24 hours. Budget ceiling is $500. No strict per-request latency SLO, but we want to maximize throughput to finish as fast as possible within budget."
This is fundamentally different from online serving. There's no per-request latency SLO. Instead you have a completion deadline, a budget ceiling, and a massive fixed workload. The system's goal flips from "minimize cost while meeting latency SLO" to "maximize throughput while staying under budget and finishing before deadline."

Module 1: The DB Scopes the Landscape
The DB’s job is the same as before, take candidate configs and return predicted metrics. But for batch inference, the relevant metrics shift. TPOT and TTFT per request still matter mechanically (they determine throughput), but the Oracle's primary output the system cares about is aggregate throughput in tokens per second and the derived estimate of total completion time and total cost.
Hard Constraint Pruning
DeepSeek-V3 is 671B total parameters, but MoE means only about 37B parameters are active per token (8 out of 256 experts plus shared attention layers). However, ALL 671B parameters must be loaded into GPU memory even though only 37B activate per forward pass. In FP16 that's roughly 1.34TB of weight memory. In FP8 that's about 671GB.
The DB starts eliminating impossible configs.
Single GPU type analysis. 671GB of weights in FP8. An A100-80GB holds 80GB. You'd need at minimum 671/80 ≈ 9 GPUs just for weights with zero room for KV cache. Realistically, you need at least 10-12 A100s to have any KV cache space. An H100-80GB is similar math. L40S at 48GB would need 671/48 ≈ 14 GPUs minimum just for weights, probably 16+ to be viable. Weights need to be accessible to the compute, so this splits across TP and PP dimensions.
TP constraints for MoE. This is where DeepSeek-V3 gets tricky and where the DB's domain knowledge matters. In a standard dense model, TP splits attention heads and FFN layers across GPUs. In an MoE model, TP also affects how experts are distributed. DeepSeek-V3 has 256 experts. With TP=8, each GPU handles 256/8 = 32 experts. With TP=4, each GPU handles 64 experts. The expert routing involves all-to-all communication between TP ranks, tokens need to reach whichever GPU holds their assigned expert. Higher TP means more communication partners and higher all-to-all overhead. This is a known pain point that the DB’s analytical model encodes.
The Oracle also checks: DeepSeek-V3's attention has 128 heads and 128 KV heads (it uses MHA, not GQA). TP must divide 128 evenly. TP = 1, 2, 4, 8, 16, 32, 64, 128 all work mathematically. But TP=1 and TP=2 can't hold the weights on A100/H100 (even in FP8, TP=2 means 335GB per GPU, far beyond 80GB). So TP must be at least 8 on 80GB GPUs. TP=16 and beyond are feasible but the all-to-all communication overhead for MoE expert routing becomes punishing.
PP analysis. DeepSeek-V3 has 61 layers. PP must divide this somewhat evenly. PP=1 (no pipeline parallelism) means all layers on every TP group — only works if TP alone provides enough memory. With TP=8 on A100, each GPU holds 671/8 ≈ 84GB of FP8 weights. That exceeds 80GB. TP=8 alone on A100 doesn't fit. You need PP=2 at minimum with TP=8 on A100 (each GPU then holds ~42GB of weights, leaving 38GB for KV cache). Or TP=16 PP=1 which fits (42GB weights per GPU) but the all-to-all communication is brutal.
On H100-80GB the memory math is similar but H100 has much higher memory bandwidth and faster interconnect (NVLink4), making higher TP more viable.
After this analysis, the Oracle produces a feasible set. Let me list the most realistic candidates:
Config A: 16× A100, TP=8, PP=2, DP=1. Each GPU holds ~42GB weights, leaving ~38GB for KV cache. This supports maybe 64-128 concurrent sequences depending on context length. Oracle predicts throughput of roughly 2,800 tokens/sec output. At this rate, 400M output tokens takes about 40 hours. Too slow — exceeds the 24-hour deadline.
Config B: 16× A100, TP=8, PP=2, DP=1, but with much larger batch sizes and aggressive batching settings. Oracle predicts throughput around 3,500 tok/s with maximum batching. Still about 32 hours. Borderline or too slow.
Config C: 32× A100, TP=8, PP=2, DP=2. Two replicas processing in parallel. Oracle predicts combined throughput around 6,500 tok/s. Completion in about 17 hours. Cost: 32 GPUs × ~$1.60/GPU-hr × 17 hours ≈ $870. Over budget.
Config D: 16× H100, TP=8, PP=2, DP=1. H100's faster memory bandwidth and compute means higher throughput per GPU. Oracle predicts roughly 5,000 tok/s. Completion in about 22 hours. Cost: 16 GPUs × ~$2.50/GPU-hr × 22 hours ≈ $880. Over budget.
Config E: 8× H100, TP=8, PP=1, DP=1. With TP=8 on H100, each GPU holds ~84GB of FP8 weights. That's slightly over 80GB. Doesn't fit. Eliminated.
Config F: 16× H100, TP=8, PP=1, DP=2. Wait — TP=8 PP=1 means 8 GPUs per replica, each holding 84GB. Doesn't fit on H100-80GB either. Eliminated.
Config G: 24× A100, TP=8, PP=2, DP=1.5 — DP must be integer. Not valid. But DP=1 with 16 GPUs and 8 spare A100s. Could the 8 spare A100s be used? Not directly in the same job (TP×PP=16 is fixed). But you could run a second smaller replica if the model could fit in 8 GPUs... it can't with TP=8 PP=1 (memory). What about a heterogeneous setup — one replica of 16 A100s (TP=8, PP=2) plus a different configuration on 8 H100s (TP=8, PP=1)? TP=8 on H100 means 84GB per GPU — doesn't fit. How about 8× H100 with FP8 and TP=8 PP=1? Still 84GB, too tight.
This is getting constrained. The Oracle identifies that DeepSeek-V3 at this scale fundamentally needs at least 16 GPUs per replica (TP=8 × PP=2) on 80GB class hardware. The question becomes: how many replicas, what GPU type, and can we meet both deadline and budget?
Config H: 16× A100 TP=8 PP=2 DP=1, with optimized vLLM settings: FP8 quantization, max_num_seqs=256, chunked prefill enabled (wait — chunked prefill with PP=2 might not work in current vLLM. The Oracle's constraint checker flags this. Disable chunked prefill, use standard prefill). Oracle predicts peak throughput ~3,800 tok/s with aggressive batching. Completion: ~29 hours. Over deadline.
Config I: 16× A100 + 16× A100 = 32× A100, TP=8, PP=2, DP=2. But we only have 16 A100s in the cluster. The resource map says 16× A100 total, 8 are currently allocated to the Qwen-72B serving job from our earlier example. Only 8× A100 are free. We can't build even one full replica on available A100s alone.
The Oracle checks the full resource map: 8× A100 free, 8× H100 free, 8× L40S free. None of these alone can form a complete DP=1 replica for DeepSeek-V3 (need 16 GPUs per replica).
Config J: A mixed approach. Use 8× H100 + 8× A100 to form one 16-GPU replica? This is problematic — TP groups must be homogeneous (all same GPU type within a TP rank). But PP stages can be different GPU types. You could do PP=2 where stage 1 is TP=8 on 8× H100 and stage 2 is TP=8 on 8× A100. This is unusual but technically feasible — the pipeline sends activations between stages, and the different compute speeds just mean the pipeline is imbalanced (the A100 stage is the bottleneck). The Oracle has no direct perf DB entry for this heterogeneous config, so confidence is low, but analytically it estimates throughput at roughly 3,000 tok/s (bottlenecked by the slower A100 PP stage). Completion: ~37 hours. Too slow.
Config K: What if we reclaim A100s? The Swap Arbiter could potentially take some A100s from the Qwen-72B serving job. But that job is running at moderate load — taking its GPUs would violate its SLO. Not viable without degrading a production endpoint.
The Oracle's landscape is constrained and it honestly reports this to Module 2. There's no configuration with currently available resources that meets both the 24-hour deadline and the $500 budget for this workload size. The feasible space has a tradeoff: meet the deadline (needs more/faster GPUs, over budget) or meet the budget (fewer GPUs, over deadline).
This is exactly the kind of situation where the LLM ensemble's reasoning becomes critical.

This is the kind of creative reasoning that no numerical optimizer would produce.

Step 4: Module 4 — The Controller Sets Up
For batch inference, Module 4's initial role is simpler than for serving. There's no dynamic load to forecast — the input is a fixed dataset. The "load" is constant: maximum throughput, fill the batches as full as possible, process rows sequentially.
The MPC still runs but the planning horizon is the full remaining job duration (~22 hours). The cost function shifts:
Instead of penalizing SLO violations (there's no per-request latency SLO), the dominant terms become:
Completion time penalty: how far are we from finishing before the 24-hour deadline? If projected completion exceeds 24 hours, heavy penalty.
Cost penalty: total GPU-hours × cost per GPU-hour.
Throughput efficiency: actual throughput vs theoretical maximum. If we're leaving performance on the table, mild penalty.
The MPC evaluates the TP=4 PP=3 config against "do nothing" (wait for more GPUs):
Config D (TP=4 PP=3, 12× A100, start now): Predicted completion in 22 hours. Cost $422. Risk: unusual config, low Oracle confidence, might be slower than predicted.
Wait option (delay 4 hours hoping H100s free up): If 8× H100 become available, could run 16× H100 TP=8 PP=2. Completion in 9 hours from that point = 13 hours from now. Cost $360. But: no guarantee H100s free up. If they don't, we've lost 4 hours and now have only 20 hours to finish — tighter deadline.
The MPC factors in the probability of H100 availability (based on current cluster state and job completion estimates) and the risk of delay. If there's a 40% chance H100s free up in 4 hours, the expected cost of waiting is: 0.4 × ($360 + 13hr timeline) + 0.6 × ($422 + 22hr timeline with 4hr delay = hitting deadline boundary). The expected cost of starting now is: $422 + 22hr timeline with full margin.
With the deadline being hard, the MPC is risk-averse about waiting. The penalty for missing the 24-hour deadline is extremely high (job failure). Starting now with 2 hours of margin is safer than gambling on H100 availability.
MPC recommends: Deploy Config D immediately.
The Gain Scheduler selects a batch-optimized regime: monitor throughput as the primary metric (not latency), long correction interval (batch jobs don't need fast reactions — throughput changes slowly), and high sensitivity to throughput degradation (if we're slower than predicted, we need to know quickly because the deadline is fixed).

Step 5: Deployment and Module 3 — Batch Monitoring
The job deploys. DeepSeek-V3 loads across 12× A100 with TP=4 PP=3. Weight loading takes about 4-5 minutes for a 671B model. The batch inference engine starts pulling rows from the dataset queue.
Module 3's monitoring for batch inference tracks different signals than for serving:
Primary metrics:
Throughput: tokens per second (output). This is the number that determines whether we meet the deadline.
Progress: rows completed out of 800K. Gives a real-time completion estimate.
GPU utilization across all 12 GPUs. If some are underutilized, the PP stages might be unbalanced.
Batch efficiency: average batch size vs configured maximum. If batches aren't filling, something is wrong.
Secondary metrics:
Per-token latency (TPOT): Not for SLO compliance but as a diagnostic. If TPOT suddenly increases, something is degrading.
Memory utilization per GPU: watching for creep toward OOM, especially with the tight 24GB KV cache headroom.
Pipeline bubble fraction: time spent idle between PP stages. This is the main efficiency concern with PP=3.
Derived metric — the critical one:
Estimated completion time = (total_output_tokens - completed_output_tokens) / current_throughput. This must stay under (deadline - current_time). Module 3 computes and tracks this continuously.

The deadband for batch jobs is configured differently than for serving. Instead of zones around a latency SLO, the zones are around the completion timeline:
GREEN: Estimated completion more than 2 hours before deadline (healthy margin).
YELLOW: Estimated completion within 2 hours of deadline (tight, need to watch).
RED: Estimated completion after deadline (going to miss it, act now).
BLUE: Estimated completion more than 6 hours before deadline (way ahead of schedule, might be overpaying).
At 22.7 hours estimated completion with a 24-hour deadline, we're 1.3 hours of margin. That's YELLOW zone — tight.
The controller wakes up. The gain scheduler sees: we're in batch mode, deadline-constrained, no immediate crisis but low margin. It selects parameters that emphasize completion assurance over cost savings.
The MPC Evaluates
The MPC looks at options:
Do nothing: Continue at 1,960 tok/s. Probably finish in 22.7 hours. Margin is thin — if throughput degrades at all due to dataset distribution shift (later rows might have longer outputs) or KV cache pressure growing, we could miss the deadline.
Tune vLLM settings: Increase max_num_seqs from 64 to 128 to allow more concurrent sequences. This might improve throughput by filling GPU time more efficiently. But it also increases KV cache memory — with only 24GB headroom, doubling concurrent sequences might push toward OOM. The Oracle predicts throughput at max_num_seqs=96: approximately 2,100 tok/s. Memory check: ~21GB KV cache per GPU, still fits but with only 3GB margin. Risky.
Adjust batch parameters: Increase max batch tokens to encourage larger batches. Similar throughput gain, similar memory risk.
Accept and monitor: The current throughput will likely get the job done. The 1.3-hour margin, while thin, is probably sufficient unless something goes wrong.
The MPC computes costs. "Tune vLLM to 96 sequences" gives a projected completion of 21.1 hours (margin improves to 2.9 hours) but introduces OOM risk. The OOM penalty is severe — if we OOM, the process crashes, we lose in-flight batches, restart takes 5 minutes, and throughput dips during recovery. Expected cost of the OOM risk, scaled by probability, versus the benefit of the extra margin.
Given that the Oracle has low confidence for this unusual config (meaning the OOM risk is hard to estimate), the MPC takes the conservative path: do nothing, monitor closely, re-evaluate in 30 minutes. If throughput holds at 1,960 tok/s, we'll be fine. If it drops, we'll act then with more information.

Step 6: Throughput Changes — The Controller Adapts
Two hours in, the monitoring picture shifts. The dataset has a non-uniform distribution — the first chunk of rows had average input length of 600 tokens, but the current chunk averages 1,100 tokens. Longer inputs mean more prefill compute per row, and the output-to-input ratio is different.
Filtered throughput drops from 1,960 tok/s to 1,650 tok/s as the engine processes longer inputs. The estimated completion time jumps to 25.2 hours. The deadline is 24 hours from start, so we now have 22 hours remaining. 25.2 > 22. RED zone.
The controller jumps past the gain scheduler directly to emergency action protocol. For batch jobs, the emergency plan is: escalate to Module 6 (Swap Arbiter) to find more resources immediately, and simultaneously check if any job parameters can be tuned without resource changes.
Module 6 activates. It scans the cluster. What's changed in the last 2 hours?
The Qwen-72B serving job is still running on 8× A100 (DP=2, overnight traffic is light at 8 users). The Swap Arbiter checks: Qwen-72B at DP=1 with 10 users — TPOT would be ~27ms, well under 35ms SLO. We could reclaim 4× A100 from its second replica. That gives us 4 more A100s.
But we need them to actually help. The current config is TP=4 PP=3 on 12× A100. Adding 4× A100 could allow:
TP=4 PP=4 on 16× A100: 61 layers / 4 ≈ 15 per stage. More PP stages means smaller bubbles per stage but more pipeline latency. Net effect on throughput is uncertain.
TP=8 PP=2 on 16× A100: The standard recommended config. TP=8 means 128/8 = 16 heads per GPU, 256/8 = 32 experts per GPU. PP=2 means 30-31 layers per stage. This is a well-studied config with high Oracle confidence.
The switch from TP=4 PP=3 to TP=8 PP=2 is a full reconfiguration. All weights must be resharded and reloaded. That's probably 5-7 minutes of downtime. During those 5-7 minutes, zero throughput. The cost is 5-7 minutes × 0 tok/s = losing about 600K-700K tokens of progress time. But the benefit is potentially much higher throughput for the remaining 20+ hours.
The Oracle predicts TP=8 PP=2 on 16× A100 for DeepSeek-V3: approximately 3,800 tok/s. That's nearly double the current 1,650 tok/s. At 3,800 tok/s, remaining output tokens (let's say 145M remaining) would take about 10.6 hours. We have 22 hours remaining. Massive margin recovery.
The Swap Arbiter computes the NBS:
Benefit: Throughput increase from 1,650 to 3,800 tok/s. Time saved: completing in 10.6 hours instead of projected 25.2 remaining hours. The entire job success depends on this swap. Benefit is "the job succeeds vs the job fails its deadline."
Cost: 7 minutes of downtime during reconfiguration. Qwen-72B serving goes from DP=2 to DP=1 (minor degradation during low-traffic period, well within SLO). Resources used increase from 12 to 16 A100s, adding cost of 4 × $1.60/hr × 10.6 hours = $68 additional.
NBS is overwhelmingly positive. The job fails without the swap and succeeds with it. This doesn't need LLM review — the numbers are unambiguous.
Action plan:
Scale Qwen-72B serving from DP=2 to DP=1 (drain one replica, ~1 minute).
Stop current DeepSeek-V3 batch processing. Checkpoint progress (record which rows are completed).
Reconfigure DeepSeek-V3: reload weights across 16× A100 with TP=8 PP=2 (~5 minutes).
Resume batch processing from the checkpoint.
Module 4's anti-windup guard activates for both jobs. The Qwen-72B controller is frozen for 5 minutes (minor reconfiguration). The DeepSeek-V3 controller is frozen for 12 minutes (major reconfiguration + settling time).
After reconfiguration, the system stabilizes. Filtered throughput on the new config: 3,650 tok/s. Slightly below the Oracle's 3,800 prediction (the longer-input rows affect throughput more than the Oracle estimated), but massively better than the previous 1,650.
New estimated completion: 145M remaining output tokens / 3,650 tok/s ≈ 11 hours. We have about 21.5 hours remaining (accounting for the 30 minutes spent on the swap process). GREEN zone — 10.5 hours of margin.
Module 5 records everything: the TP=4 PP=3 config's actual throughput at various input lengths (crucial data for a rare config), the throughput improvement from switching to TP=8 PP=2, the transition cost (7 minutes downtime + checkpoint overhead). The CatBoost model gains data points for both configs. The policy memory records the entire episode as a lesson: "for DeepSeek-V3 batch processing with variable input lengths, TP=4 PP=3 underperforms especially on long inputs. TP=8 PP=2 is strongly preferred when 16 GPUs are available. The throughput difference is roughly 2× and justifies the reconfiguration cost."

Step 7: Smooth Sailing — The Monitor Watches
For the next 8 hours, the batch job processes rows steadily. Throughput fluctuates between 3,400 and 3,800 tok/s depending on input length distribution in the current batch. The Kalman filter smooths this to a reliable trend line.
The deadband stays GREEN throughout. The controller is dormant. The MPC doesn't run. The anti-windup guard is inactive.
Module 5 quietly accumulates data points. Every 10 minutes, it records the current throughput, the Oracle's prediction for the current batch characteristics, and the residual. The CatBoost correction model is building a rich picture of how DeepSeek-V3 throughput varies with input length, batch size, and GPU configuration on A100s.
At hour 15, the batch job has completed 650K out of 800K rows (the hard rows on DeepSeek-V3). Remaining: 150K rows, approximately 30M output tokens. At 3,650 tok/s, that's about 2.3 more hours. Total completion around hour 17. The deadline is hour 24. Massive margin.
Meanwhile, the Qwen-72B easy rows (sub-job Alpha) have been processing on the Qwen-72B serving endpoint during overnight hours. With low serving traffic, the batch requests get scheduled during idle cycles. 1.2M easy rows at roughly 500 tok/s batch throughput during off-peak gives about 480K rows done overnight (12 hours). The remaining 720K easy rows will need to wait for the next off-peak window or process during daytime at lower priority.
Actually, this is a problem — the easy rows are taking longer than expected because they compete with serving traffic during daytime. Module 3 flags this: sub-job Alpha is behind schedule.
But the overall job (Alpha + Beta) can be restructured. Since Beta (DeepSeek-V3 hard rows) will finish early with massive margin, and we'll be freeing 16 A100s in a few hours, we can dedicate some of those freed A100s to running Qwen-72B batch processing at full speed to catch up on the easy rows. The Swap Arbiter will handle this resource reallocation when Beta completes.

Step 8: Completion and Module 5 — Final Learning
At hour 17, sub-job Beta (DeepSeek-V3 hard rows) completes. All 800K hard rows processed, 160M output tokens generated. The DeepSeek-V3 deployment is shut down, freeing 16× A100.
The Swap Arbiter immediately evaluates: the Qwen-72B serving job should be restored to DP=2 (daytime traffic is ramping up). That takes 4× A100. The remaining 12× A100 can be allocated to sub-job Alpha's remaining easy rows, running Qwen-72B batch processing as a dedicated job.
With 12× A100 dedicated to Qwen-72B batch processing (TP=4, DP=3), throughput is high. The remaining 720K easy rows process in about 3 hours.
At hour 20, the entire 2M-row job is complete. 4 hours under deadline. Total cost: roughly $307 for the DeepSeek-V3 portion + $95 for the Qwen-72B batch GPU time + negligible marginal cost for the overnight serving-based processing. Total ~$402. Under the $500 budget.

Step 9: The Swap Arbiter Post-Completion — Resource Rebalancing
With the batch job complete, the Swap Arbiter does a global rebalance. The cluster state is now:
Qwen-72B serving: 8× A100 (TP=4, DP=2), restored to full capacity.
8× A100 free (released from batch processing).
8× H100 free (were never used for this job).
8× L40S free (too small for DeepSeek-V3).
The Arbiter checks: are there queued jobs? Is the Qwen-72B deployment right-sized for current traffic? Could any running job benefit from additional resources?
If daytime traffic on the Qwen-72B serving endpoint is moderate (50 users), the current DP=2 config is handling it with TPOT around 25ms (28% headroom on 35ms SLO). The Arbiter asks: could Qwen-72B run on cheaper GPUs? What if we moved it to L40S? The Oracle checks: Qwen-72B on 8× L40S with TP=4... memory is tight (72B FP16 / 4 = 36GB per GPU on 48GB cards = 12GB for KV cache, which limits concurrent users). At 50 users, KV cache might not fit. Oracle predicts OOM risk. The Arbiter drops this idea.
What about freeing H100s for another job while keeping Qwen-72B on A100s? H100s are the most expensive and scarce resource. If no job needs them, they should sit idle (costing nothing if not running) rather than being allocated unnecessarily. The Arbiter recommends: keep current Qwen-72B config, mark A100s and H100s as available for new jobs.
This is a simple case — no swap needed, just clean up. The Arbiter's value shows more in the mid-job resource contention scenarios like the one we saw during the batch processing crisis.

Step 10: The Circuit Breaker — What If Something Had Gone Wrong
Let's rewind to the moment after the TP=8 PP=2 reconfiguration at hour 2.5. What if, instead of stabilizing at 3,650 tok/s, the system had crashed?
Say the weight reload completes, vLLM starts, and immediately OOMs. The FP8 weight estimation was slightly off (FP8 has some runtime overhead for dequantization buffers that the Oracle didn't account for), and with 16 GPUs the per-GPU memory is slightly tighter than predicted.
The circuit breaker catches this within 30 seconds. OOM is a catastrophic signal — immediate rollback.
The circuit breaker's actions:
Record the failure: Config "TP=8 PP=2, 16× A100, FP8, max_num_seqs=64" for DeepSeek-V3 → OOM.
Add to blacklist.
Feed back to hard constraint validator: the memory estimation for DeepSeek-V3 FP8 is wrong. Increase the safety margin for this model from 85% to 80% of GPU memory. Recalculate: at 80% margin, 16× A100 TP=8 PP=2 now fails the memory check (the extra dequantization buffer pushes it over). Future attempts at this config will be caught by hard constraints.
Rollback: restart with the previous config (TP=4 PP=3 on 12× A100).
The anti-windup guard freezes the controller for 15 minutes while the rollback happens and stabilizes.
Meanwhile, the controller is now in RED zone (we were already in a deadline crisis when we attempted the reconfiguration). After the rollback stabilizes, the MPC runs with updated constraints. The TP=8 PP=2 config is now blacklisted. Options are:
Continue with TP=4 PP=3 on 12× A100 at 1,650 tok/s (will miss deadline).
Try TP=4 PP=2 on 8× A100 (uses fewer GPUs but might have higher per-GPU throughput due to less PP overhead). Oracle predicts... memory check first: 671GB FP8 / 8 = 84GB per GPU. Doesn't fit. Eliminated.
Try TP=4 PP=4 on 16× A100 (uses all GPUs, 4 PP stages). Oracle predicts: more PP stages = more pipeline bubble, but each stage is smaller. Estimated throughput ~2,800 tok/s. Memory per GPU: 671/16 = 42GB weights + KV cache ≈ 60GB total. Fits at 80% safety margin (64GB). This passes hard constraints including the newly tightened margin.
The MPC evaluates TP=4 PP=4: another full reconfiguration (7 minutes downtime), but throughput improves from 1,650 to an estimated 2,800. At 2,800 tok/s, remaining tokens would complete in about 14.4 hours. We have about 20 hours remaining. Margin restored to 5.6 hours. The swap cost is painful but the benefit is clear.
The MPC recommends the reconfiguration. It executes. This time the circuit breaker watches carefully — the config is similar to one that just OOMed (same model, same GPU count) so it's in heightened monitoring mode. But TP=4 PP=4 distributes weights differently (42GB per GPU instead of the 84GB attempt), and it stabilizes successfully.
Module 5 records the entire failure-and-recovery sequence. The CatBoost model gets an invaluable negative data point. The policy memory records: "DeepSeek-V3 FP8 on A100 at TP=8 PP=2 OOMs due to dequantization buffer overhead. TP=4 PP=4 on 16× A100 is the correct fallback at ~2,800 tok/s." Future LLM ensemble proposals for DeepSeek-V3 will include this lesson.
